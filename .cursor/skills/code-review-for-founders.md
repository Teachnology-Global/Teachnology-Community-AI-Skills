# Code Review for Non-Technical Founders

## Purpose

You don't need to read code to spot problems in it. This skill gives you a structured checklist to review AI-generated or developer-written code — focusing on what matters: "Is this safe? Is it complete? Does it do what we asked?"

**For founders and teachers:** If an AI or developer hands you finished code, don't just click "approve." Use this checklist every time. It takes 5 minutes and catches 80% of the problems that cause real damage.

## Activation

This skill activates when you mention:

- An AI agent reports "task complete" and asks for review

## The 5-Minute Review Checklist

### ✅ Step 1: What Was Meant to Happen?

**Ask yourself (or the AI/developer):**

1. What was the original feature request?
2. What files were changed?
3. How many lines changed? (Large PRs = more risk)

**Red flags:**
- ⚠️ "I also improved some other things" — scope creep in code
- ⚠️ More than 50 files changed for a simple feature
- ⚠️ Files touched that don't relate to the feature
- ⚠️ No tests included

**What to say if you see these:**
> "Please limit this PR to only the changes needed for [specific feature]. Any additional improvements should be separate PRs."

### ✅ Step 2: Does It Actually Work?

**Questions to ask (even without reading code):**

1. "Can you show me the feature working?" (Ask for a screenshot, demo, or video)
2. "What happens when [edge case]?" Examples:
   - What if a user enters negative numbers?
   - What if the file is empty?
   - What if the network is slow or down?
   - What if someone has a really long name/email/phone?

**Red flags:**
- ⚠️ "The happy path works" but no mention of error cases
- ⚠️ Cannot demonstrate the feature
- ⚠️ Only tested locally, no staging environment

**What to say:**
> "Before I approve, I need to see: (1) the feature working end-to-end, and (2) how it handles [specific error case]."

### ✅ Step 3: Security Quick Check

**You don't need to read the code for this. Just ask:**

1. "Does this feature handle any passwords, API keys, or user data?"
2. "Where are the secrets stored?" (Answer MUST be "environment variables" or "secrets manager" — never "in the code")
3. "Can someone do more than they're supposed to?" (Authorization)
4. "Can someone send malicious input?" (Input validation)

**Red flags:**
- ⚠️ API keys, passwords, or tokens visible in any code file
- ⚠️ "The user can just type anything" into a field
- ⚠️ Any file path or URL that mentions "localhost", "dev", or "test"
- ⚠️ Database queries that look like `SELECT * FROM users WHERE id = ` + `user_input`

**What to say:**
> "I see [specific concern]. Please confirm: (1) secrets are in environment variables only, (2) user input is validated on the server side, (3) we test with malicious input before merging."

### ✅ Step 4: Completeness Check

**A feature isn't done until it's done. Ask:**

1. "Are there tests for this?" (Minimum: 1 test per major use case)
2. "Is there a changelog entry?" (What changed and when)
3. "Is the documentation updated?" (README, setup guide, API docs)
4. "What happens if this breaks in production?" (Monitoring/alerting plan)

**Red flags:**
- ⚠️ "I'll add tests later" — they won't
- ⚠️ No mention of monitoring or error handling
- ⚠️ Feature depends on another unmerged feature
- ⚠️ Database changes without a migration plan

**What to say:**
> "This isn't ready until: (1) tests exist and pass, (2) changelog is updated, (3) we know how to detect when it breaks."

### ✅ Step 5: The Regression Check

**The most common mistake: fixing one thing and breaking another.**

**Ask:**
1. "What existing features could this affect?"
2. "Did you run the full test suite, not just new tests?"
3. "Are there breaking changes to the API?"

**Red flags:**
- ⚠️ "I only tested the new feature" 
- ⚠️ "Shouldn't affect anything else" (without proof)
- ⚠️ Changes to core/shared files (auth, database, billing)

## Decision Matrix

| Checklist Result | Action |
|-----------------|--------|
| All steps ✅ | **Approve** with a brief comment |
| 1-2 minor items missing | **"Almost"** — specify exactly what's needed, then approve when done |
| 3+ items or any security concern | **"Not yet"** — reject and request revisions with clear feedback |
| Cannot determine scope or safety | **Escalate** — bring in a technical advisor or the community |

## What to Write in Your Review

### When Approving:
```
✅ This looks good. Specifically:
- Addresses the original requirement for [feature]
- Tests cover the main scenarios
- No secrets exposed
- Clean, focused changes

Approved. Ship it.
```

### When Asking for Revisions:
```
⚠️ Needs revision before merge:

1. [Specific issue] — Please [what to do]
2. [Specific issue] — Please [what to do]

I'll re-review when these are addressed. 
```

### When Rejecting:
```
❌ Cannot approve as-is:

[Clear one-paragraph summary of why]

Key concerns:
- [Security concern if any]
- [Completeness concern]
- [Scope concern]

Please revise and resubmit. Happy to discuss the approach if you'd like feedback before rebuilding.
```

## Common AI Code Review Traps

### Trap 1: "It looks professional, so it must be right"
AI is excellent at writing code that *looks* correct but has subtle logic bugs. Always verify the actual behaviour, not the code style.

### Trap 2: "It works on my machine"
Development environment success ≠ production readiness. Always ask about database differences, environment variables, and deployment configs.

### Trap 3: "It's a small change, can't cause problems"
Small changes to authentication, billing, or database schemas can cause massive damage. These always need extra scrutiny regardless of size.

### Trap 4: "I'll approve this one, it's from a trusted source"
Trusted sources write bugs too. Every code change needs the checklist, even from people who've been right 99 times before.

### Trap 5: "The AI explained it, so I understand it"
An AI can explain complex code in simple words, but that doesn't mean the code is correct. Understanding the explanation ≠ verifying the implementation.

## Quick Reference Card

Print this. Use it every time you see a PR.

```
CODE REVIEW — 5 MINUTES
1. ✅ Match to original request? (files, scope, no extra changes)
2. ✅ Tested and demonstrated? (works, handles edge cases)
3. ✅ No secrets? (passwords, keys, tokens NOT in code)
4. ✅ Complete? (tests, changelog, docs, monitoring plan)
5. ✅ No regressions? (other features safe, full test suite passed)

Result: APPROVE | REVISE | REJECT | ESCALATE
```

## Notes for Community Mentors

When teaching non-technical founders to review code:

1. **Start with Step 1** — teach them to define what "done" looks like before looking at what was delivered.
2. **Use the red flags** — beginners are good at spotting things that look wrong even if they can't explain why technically.
3. **Provide templates** — the "What to say" scripts above give beginners confidence to ask the right questions.
4. **Never approve on their behalf** — they need to build the muscle. Guide, don't do it for them.
5. **Celebrate rejections** — rejecting bad code is a bigger win than approving good code. Train this.
